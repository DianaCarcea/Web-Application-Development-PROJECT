1. Abstract

This technical report presents a Semantic Web–based platform for the exploration and management of provenance information related to artistic works from Romania’s cultural heritage. The system integrates open cultural data provided by Europeana with complementary structured knowledge retrieved from Wikidata, including information about artists, artworks, and cultural institutions. By relying on RDF data modeling, SPARQL querying, and Linked Data principles, the platform enables semantic integration of heterogeneous sources and supports advanced functionalities such as contextual navigation, structured visualization, and data-driven exploration of artworks. The proposed solution aims to enhance accessibility, interoperability, and reuse of open cultural heritage data, contributing to its educational, social, and research-oriented value.

2. Introduction

The digitization of cultural heritage has led to the publication of large volumes of open data by cultural institutions and international initiatives such as Europeana and Wikidata. These platforms provide public access to information about artworks, artists, and museums, supporting research, education, and cultural dissemination. As a result, cultural heritage data has become increasingly available, but also highly fragmented across multiple sources and formats.

Although such datasets are openly accessible, they are often published in heterogeneous representations and with limited semantic alignment. This fragmentation restricts the ability to explore cultural heritage in an integrated manner and makes it difficult to analyze relationships such as artwork provenance, artistic movements, or institutional context. Addressing these challenges requires approaches capable of integrating distributed datasets and enabling meaningful semantic connections between them.

Semantic Web technologies offer a standardized framework for representing, interlinking, and querying cultural heritage data. Through the use of RDF, shared vocabularies, and SPARQL, heterogeneous cultural datasets can be integrated into coherent knowledge graphs that support complex exploration scenarios and reuse across applications.

3. Problem statement

Despite the availability of open cultural heritage data, existing platforms exhibit important limitations when it comes to integrated exploration of artwork provenance. Europeana provides access to Romanian cultural heritage datasets primarily in XML-based formats, where information about artists, institutions, and provenance relationships is often incomplete or weakly structured. Conversely, Wikidata offers rich, well-structured knowledge about artists and cultural entities, but does not comprehensively cover national heritage collections.

The lack of semantic interoperability between these datasets hinders unified exploration and limits users’ ability to obtain a coherent view of artworks within their cultural, historical, and institutional context. As a consequence, users are required to navigate multiple disconnected platforms, reducing both usability and the potential for meaningful data reuse

4. Proposed solution and social impact

This project proposes a web-based platform that semantically integrates Romanian cultural heritage artworks sourced from Europeana with complementary data retrieved from Wikidata. The collected data is transformed into RDF and modeled using established semantic vocabularies, enabling SPARQL-based access and semantic navigation across artworks, artists, and institutions.

The platform aims to deliver social and educational impact by improving access to structured cultural knowledge and by supporting contextual exploration of artistic heritage. Through a user-oriented interface and features such as artwork-centric navigation and recommendations, the system encourages cultural discovery and facilitates the reuse of open cultural data.

5. System Architecture and API Design

This chapter details the architectural design of the ArP Digital Catalog, utilizing the C4 Model to visualize the system's interactions, containers, components, and data structures. The architecture is designed to function as a semantic aggregator, orchestrating data flow between external open-data sources and end-users through a hybrid web interface.

5.1 System Context (Level 1)

The System Context Diagram illustrates the high-level landscape of the application. It highlights the boundary between the internal ArP Environment and the External Data Sources required for data enrichment.

The system serves two distinct types of actors:

  • Web Visitor: A human user accessing the system via a browser to discover artists and artworks through the graphical interface (HTML).
  • Developer / API Client: A technical consumer accessing the system programmatically to retrieve structured JSON data or execute direct SPARQL queries.

To fulfill its role, the ArP Digital Catalog integrates with several External Systems:

  • Wikidata API: Used via SPARQL to extract metadata for international artworks and fill gaps in local Romanian records.
  • Getty Vocabularies: Accessed to standardize technical terms (materials, categories) using AAT, ULAN, and TGN identifiers.
  • Wikimedia Commons & Google Images: Utilized for resolving and scraping visual assets (images) for the cataloged items.

5.2 System Containers (Level 2)

The Container Diagram zooms into the ArP Ecosystem, differentiating between the runtime web application and the offline data processing tools.

The ecosystem is composed of four main containers:

  • Spring Boot Application: The core Java 21 monolith. It handles business logic, performs server-side rendering (Spring MVC), and exposes the REST API.
  • RDF File Storage: The persistent storage layer consisting of static Turtle (.ttl) files (e.g., artworks_arp.ttl, artists.ttl). These files act as the reliable source of truth.
  • ETL & Ingestion Modules: A set of standalone Java console applications (LidoToTtlConverter, GettyFetchers). These tools transform raw local XML (LIDO) and remote SPARQL data into the standardized RDF files.
  • In-Memory Knowledge Graph (Apache Jena): A high-performance semantic graph. It is initialized at startup by reading the RDF File Storage, enabling fast, read-only SPARQL operations during runtime.

5.3 Backend Components (Level 3)

To support the dual requirement of serving both humans and machines, the backend adopts a Hybrid Architecture. The Component Diagram below illustrates the internal layering of the Spring Boot Application.

The internal components interact as follows:

5.3.1 Presentation Layer (Hybrid)

  • Web Controllers: (e.g., ArtworkController) Serve HTML views using Thymeleaf. They handle page loads and deliver HTML fragments for AJAX requests (e.g., recommendation sliders).
  • API Controllers: (e.g., ApiArtistController, SparqlController) Bypass the view resolver to serve raw JSON and handle binary exports (ZIP).

5.3.2 Business & Data Access Layer

  • Repositories: (ArtworkRepository) The data access layer. Unlike traditional ORMs, these components load .sparql templates, execute them against the Apache Jena Dataset, and manually map the resulting ResultSet into Domain POJOs.
  • Configuration: (RDFConfig, PredefinedQueryLoader) Responsible for bootstrapping the application by loading the TTL files into memory at startup.

The REST API is organized around resource-oriented controllers using standard HTTP verbs. A global filtering parameter, domain (ro/int), is consistently implemented across endpoints.

The system implements the OpenAPI 3.0 Specification using springdoc-openapi. The API contract is generated dynamically at runtime by introspecting controller annotations (@Tag, @Operation). An interactive Swagger UI interface is automatically exposed at /swagger-ui.html for testing.

6. Internal Data Structures and Application Models

The application implements a data model strictly aligned with the W3C PROV-O (Provenance Ontology) standards. Instead of an arbitrary database schema, the internal Java classes mirror the semantic triplets used in the Knowledge Graph: Entities (what was created), Agents (who influenced the entity), and Activities (how the entity was modified). This alignment ensures a seamless transition between the RDF layer and the Object-Oriented application logic.

6.1 Domain Model Overview

The domain model is built upon three abstract base classes that enforce a generic structure across the system:

  • Entity: Represents physical or digital artifacts. In the context of this project, it serves as the parent for the Artwork class.
  • Agent: Represents entities that bear responsibility for an activity. This is the superclass for Museum and Person. The Person class is further specialized into Artist, Registrar, Validator, and Collector.
  • Activity: Represents dynamic events that affect the entity's state. The primary implementation used is Creation, which captures the temporal and technical context of an artwork's origin.

6.2 Java Application Models

The domain entities are implemented as Java POJOs (Plain Old Java Objects). The main Data Transfer Object (DTO) is the Artwork class, which extends the generic Entity class. This structure allows the application to handle data uniformly across the view layer while preserving the specific attributes required for detailed views.

Key implementation details include:

  • Inheritance Strategy: The Java classes utilize inheritance to avoid code duplication. For instance, Artist extends Person, which in turn extends Agent, inheriting standard fields like URI and Name.
  • Complex Types: The ownershipHistory is modeled as a List<Collector>, where each Collector object stores the owner's name, start date, and end date of the possession.
  • Flexibility: Fields like materialsUsed or techniques are stored as List<String> to accommodate the variability of cultural metadata retrieved via SPARQL GROUP_CONCAT functions.
  • Unified Data Representation: The model acts as a hybrid aggregator, encapsulating fields specific to the local heritage (e.g., cimecLink) alongside fields retrieved from external sources (e.g., wikidataLink).

Below is a snippet of the central Artwork class, illustrating the aggregation of related agents and activities: 

public class Artwork extends Entity {
    public String inventoryNumber;
    public String category;

    // Specific external and internal identifiers
    public String cimecLink;
    public String wikidataLink;

    // Aggregation of related entities
    public Creation creation;
    public Registrar registrar;
    public Validator validator;

    // Handling multi-valued properties and provenance
    public List<String> materialsUsed;
    public List<Collector> ownershipHistory = new ArrayList<>();
}

6.3 Mapping RDF Data to Application Objects

Since the application interacts with a Semantic Knowledge Graph rather than a relational database, standard ORM (Object-Relational Mapping) frameworks are not applicable. Instead, the system implements a dual manual mapping strategy to handle data retrieval from both the local Jena Knowledge Graph and the remote Wikidata SPARQL endpoint.

6.3.1 Local Repository Mapping (Structured Parsing)

In the ArtworkRepository, the mapping logic transforms semantic triples into Java objects by iterating through the QuerySolution results. To overcome the limitation of SPARQL SELECT queries returning flat tabular data, the application implements specific parsing logic:

  • List Aggregation: Multi-valued properties (e.g., classifications, materials) are retrieved as concatenated strings using the SPARQL GROUP_CONCAT function. The Java application splits these raw strings (delimited by ;) to populate List<String> fields.
  • Provenance Reconstruction: The most complex mapping task involves the ownership history. The query retrieves provenance events as composite strings in the format "OwnerName||StartDate||EndDate". The application parses these tokens programmatically to instantiate Collector objects, handling null values and sorting the history chronologically in memory.

6.3.2 Wikidata Federated Mapping (Data Normalization)

The WikidataArtworkRepository interacts with the live Wikidata SPARQL service. The mapping layer here includes extensive Data Normalization logic to handle the heterogeneity of crowd-sourced data:

  • Date Standardization: Wikidata dates (`inception`, `pointInTime`) vary in format (ISO timestamps vs. simple years). The repository implements Regex-based parsing to extract the year uniformly.
  • Dimensional Parsing: Physical dimensions are often returned as separate values and units (e.g., "50" and "cm"). The mapping logic concatenates these into a human-readable dimensions string.
  • Advanced Image Orchestration: Standard queries often yield missing or incompatible image URLs (e.g., TIFF files). The application employs an ImageOrchestrator with a "Waterfall" discovery strategy:
    • It prioritizes the Wikipedia PageImages API for accuracy, falling back to a Google Images Scraper and Wikimedia Commons search if necessary.
    • Transcoding: It detects unsupported formats (like .tif) and dynamically rewrites the URL (appending width=800) to force Wikimedia to serve a browser-compatible JPG thumbnail.

The following code snippet demonstrates the parsing logic used to reconstruct the provenance chain from a raw SPARQL result: 

// Extracting and parsing the Ownership History from a flat SPARQL string
  if (sol.contains("ownershipHistory")) {
      String raw = sol.getLiteral("ownershipHistory").getString();

      for (String entry : raw.split(";;")) {
          // Splitting the composite token "Name||Start||End"
          String[] parts = entry.split("\\|\\|", -1);

          Collector o = new Collector();
          o.name = parts.length > 0 ? parts[0] : "?";
          o.startedAt = parts.length > 1 && !parts[1].isEmpty() ? parts[1] : "?";
          o.endedAt = parts.length > 2 && !parts[2].isEmpty() ? parts[2] : "?";

          artwork.ownershipHistory.add(o);
      }
      // Sorting logic to ensure chronological order
      artwork.ownershipHistory.sort(comparator);
  }

6.4 Repository and Service Layer Design

The system architecture strictly follows the Service-Repository pattern facilitated by the Spring Boot framework. This design ensures a clear separation of concerns: the Controller layer handles HTTP requests, the Service layer manages business logic and model selection, and the Repository layer encapsulates the complexity of Semantic Web technologies.

6.4.1 The Repository Layer (Semantic Data Access)

The application defines specialized repositories (e.g., ArtworkRepository, WikidataArtworkRepository) that abstract the underlying data sources. A key architectural decision was the externalization of SPARQL queries. Instead of hardcoding complex SPARQL strings within Java classes, the system loads them dynamically from external .sparql resources (e.g., /sparql/artwork-find-all.sparql). This approach enhances maintainability, allowing queries to be tested and optimized independently of the compilation process.

The repositories implement two distinct access strategies:

  • In-Memory Graph Access: For the Romanian Heritage dataset, the repository queries a local Apache Jena Model loaded at startup.
  • Federated Querying (SPARQL Client): For the International dataset, the WikidataArtworkRepository acts as an HTTP client, constructing and executing requests against the public https://query.wikidata.org/sparql endpoint using QueryExecutionHTTP.

6.4.2 The Service Layer (Orchestration & Context Switching)

The Service Layer (represented by ArtworkService and WikidataArtworkService) acts as a facade, orchestrating data flow and handling Model Selection.

A distinctive feature of the business logic is the dynamic Context Switching based on the user's intent. Methods such as getHomepageArtworks accept a domain parameter. Based on this value (`"ro"` or `"int"`), the service routes the request to the appropriate repository and data model. This abstraction allows the Controller to remain agnostic of the underlying data source, processing lists of Artwork DTOs uniformly regardless of whether they originated from a local ontology or the global Wikidata knowledge graph.

Additionally, this layer handles presentation logic such as:

  • Pagination: Calculating SPARQL LIMIT and OFFSET based on page numbers.
  • Contextual Recommendations: The system encourages exploration by suggesting artworks that share:
    • Authorship: Other works created by the same Artist.
    • Institutional Context: Artworks housed in the same Museum or location.
    • Thematic Classification: Artworks falling under the same Category.

7. RDF Knowledge Modeling and Semantic Vocabularies

The core of the developed system is a robust semantic data model capable of representing the complexity of artworks, their historical context, and provenance relations. The modeling strategy follows a hybrid approach: reusing standard W3C vocabularies for interoperability and defining a custom domain-specific ontology (ARP) to capture the particularities of the Romanian national heritage.

7.1 Adopted Semantic Vocabularies

To ensure integration into the Linked Open Data (LOD) ecosystem, the system extensively uses standardized prefixes and definitions:

  • PROV-O (The PROV Ontology): The backbone of the model, used to describe the lineage, creation, and custody transfer of artworks. Prefix: prov: <http://www.w3.org/ns/prov#>.
  • Dublin Core Terms (DCTerms): Used for general descriptive metadata, such as licensing (dct:license) and identifiers. Prefix: dct: <http://purl.org/dc/terms/>.
  • FOAF (Friend of a Friend): Used for modeling agents (people, organizations). Prefix: foaf: <http://xmlns.com/foaf/0.1/>.
  • RDFS & OWL: The fundamental languages used for defining classes, properties, and ontological hierarchies.

7.2 The Custom ARP Ontology

Since generic standards do not fully cover the specific workflow of the Romanian museum domain (e.g., the specific role of a "Registrar" in cataloging, the distinction between private collectors and public museums), a custom ontology named ARP (Artwork Provenance) was defined.

The ontology extends the three core classes of the W3C PROV standard: Entity, Agent, and Activity, using the inheritance mechanism (rdfs:subClassOf). The defined namespace is http://arp.ro/schema#.

Class Hierarchy and Semantics

The following definitions illustrate how the custom classes map to the standard PROV model, explaining the specific purpose of each derived concept within the system: 


# =========================================================
# 1. ENTITIES (Digital or Physical Objects)
# Base Class: prov:Entity
# =========================================================

    rdfs:subClassOf prov:Entity ;
    rdfs:comment "The core entity of the system. Represents the physical cultural heritage object (painting, drawing, etc.)." .

arp:Collection a owl:Class ;
    rdfs:subClassOf prov:Entity ;
    rdfs:comment "A logical grouping of artworks (e.g., 'The Royal Collection'). Can act as a container for provenance context." .


# =========================================================
# 2. AGENTS (Who is responsible?)
# Base Class: prov:Agent
# =========================================================

# --- Intermediate Base Classes ---
arp:Agent a owl:Class ;

    rdfs:label "Agent" .

    rdfs:label "Person" ;

7.3 Data Transformation and Resource Identification

The knowledge graph is populated through a dual-channel ETL (Extract-Transform-Load) process, consolidating data from heterogeneous sources into a unified RDF format.

A. Ingestion from National Heritage (LIDO XML)

The primary dataset, artworks_arp.ttl, is generated by parsing legacy LIDO XML files provided by CIMEC via Europeana. The transformation logic is encapsulated in LidoToTtlConverter.java, which maps XML tags to ARP ontology properties.

B. Ingestion from Wikidata (SPARQL)

To enrich the system with international context, a secondary dataset artworks_arp_int.ttl is generated using SparqlToTTLArp.java. This component:

  1. Executes SPARQL queries against the Wikidata Query Service to fetch popular artworks (wd:Q838948).
  2. Extracts metadata (labels, dimensions, materials, significant events).
  3. Programmatically constructs an RDF model using Apache Jena, ensuring the resulting Turtle file adheres strictly to the ARP ontology structure.

C. Resource Identification Strategy (URI Policy)

Regardless of the source (LIDO or Wikidata), the system enforces a strict URI policy to ensure global uniqueness and dereferenceability within the http://arp.ro/resource/ namespace. The implementation sanitizes input strings to generate valid IRIs.

Entity Type URI Pattern Description
Artwork http://arp.ro/resource/artwork/{ID} Unique ID (CIMEC RecID or Wikidata Q-ID).
Agent http://arp.ro/resource/agent/{Sanitized_Name} Uniform ID for Artists, Museums, and Organizations.
Activity http://arp.ro/resource/activity/creation_{Artwork_ID} Synthesized ID linking the creation event to the artwork.

7.4 Provenance Modeling with PROV-O

A distinct feature of this project is the explicit modeling of provenance using the W3C PROV-O standard. Instead of treating an artwork as a static object with simple string attributes, the system models it as the output of a dynamic process (Activity) performed by an Agent. This approach enables the representation of complex institutional hierarchies and creation contexts using real data extracted from the national heritage records.

Defining the Agents

Before establishing relationships to artworks, the agents involved (creators, custodians, administrative bodies) are instantiated with specific ARP classes. The following snippet from the generated artworks_arp.ttl dataset illustrates the distinction between an individual Artist, a Museum, and an administrative Organization:

7.5 Expressiveness and Validation (SHACL)

To guarantee the consistency and quality of the RDF data, a validation mechanism was implemented using SHACL(Shapes Constraint Language) . Unlike rigid XML schemas, SHACL allows for logical graph validation.

Defining Constraints (Shapes)

In the file arp-shapes.ttl, strict rules were defined for system entities. For example, an Artist must have a name, and a Creation must be associated with at least one Artwork and one Artist. 


arp:ArtworkShape
    a sh:NodeShape ;
    sh:targetClass arp:Artwork ;
    sh:property [
          sh:path arp:title ;
          sh:datatype xsd:string ;
          sh:minCount 1 ;
          sh:message "An artwork must have exactly one title." ;
      ] ;
    sh:property [
          sh:path arp:currentLocation ;
          sh:or ( [ sh:class arp:Museum ] [ sh:class arp:Collector ] ) ;
          sh:message "Location must be a Museum or a Collector." ;
      ] .

Java Validation Implementation

The system includes an automated validation component (ValidateRDF.java) utilizing the Apache Jena SHACL library. This component loads the data models (both the Romanian dataset artworks_arp.ttl and the international dataset artworks_arp_int.ttl) and verifies them against the defined shapes. 


// Snippet from ValidateRDF.java
Model artworkModel = rdfConfig.artworkModel(); // Loads artworks_arp.ttl
Model shapesModel = RDFDataMgr.loadModel("src/main/resources/rdf/arp-shapes.ttl");

ValidationReport report = ShaclValidator.get().validate(shapesModel.getGraph(), artworkModel.getGraph());

System.out.println("Artwork Model Conforms? " + report.conforms());
if (!report.conforms()) {
    report.getEntries().forEach(System.out::println);
}

This approach ensures that data integrated from external sources (Wikidata, CIMEC) complies with the internal model before being exposed through the SPARQL API.

8. Integration of External Knowledge Sources and Linked Data Principles

To transition from a siloed dataset to a truly semantic application, the system integrates robustly with the Linked Open Data (LOD) cloud. This is achieved not merely by manual linking, but through automated ingestion pipelines that reconcile local entities with global authorities like Wikidata and the Getty Vocabularies (AAT, ULAN, TGN).

8.1 Integration with Wikidata

Wikidata serves as the primary hub for enriching the knowledge graph with international context. The system implements a dedicated ETL process via the SparqlToTTLArp.java component to ingest popular artworks and their provenance history.

The Ingestion Pipeline

The integration logic follows a specific workflow to map Wikidata's generic structure to the custom ARP ontology:

  1. Selection: The system executes a federated SPARQL query to identify the top 100 most referenced artworks (wd:Q838948) on Wikidata.
  2. Metadata Extraction: It extracts granular details including dimensions, inventory numbers, and deep ownership history.
  3. Ontology Mapping:
    • Wikidata Entities -> arp:Artwork
    • Property wdt:P170 (creator) -> prov:wasAttributedTo
    • Property wdt:P127 (owned by) -> parsed into arp:TransferOfCustody events.

Provenance Reconstruction

A key technical achievement is the reconstruction of the ownership chain. Wikidata often stores owners as a list. The processOwnershipHistory method in the Java fetcher iterates through these qualifiers to generate distinct, time-bound provenance events: 


// Example of generated provenance from Wikidata ingestion
<http://arp.ro/resource/artwork/Q12418> arp:hasOwnership [
    a arp:TransferOfCustody ;
    prov:wasAssociatedWith <http://arp.ro/resource/agent/Collector_Napoleon> ;
    arp:startedAtTime "1797-01-01"^^xsd:date ;
    arp:endedAtTime "1815-01-01"^^xsd:date
] .

8.2 Getty Vocabulary Integration (AAT, ULAN, TGN)

To ensure semantic interoperability and standardize terminology, the system aligns its controlled vocabularies with the Getty Research Institute standards. This is critical for cross-language retrieval (e.g., searching for "wood" retrieves "lemn").

A. Concept Alignment (Materials & Categories) - Getty AAT

The components GettyCategoryFetcher.java and GettyMaterialsFetcher.java implement a bridge between Romanian labels and the Art & Architecture Thesaurus (AAT).

  • Mechanism: The system extracts Romanian terms (e.g., "ulei", "litografie") from the local dataset.
  • Translation & Reconciliation: It uses Wikidata as a pivot to translate terms to English and retrieve the corresponding Getty AAT ID property (wdt:P1014).
  • SKOS Linking: The result is a SKOS mapping file (getty-materials.ttl) linking local URIs to global concepts. 

<http://arp.ro/material/lemn>
    rdfs:label "lemn"@ro ;
    rdfs:label "wood"@en ;
    skos:exactMatch aat:300011914 .

B. Agent Reconciliation (Museums & Artists) - Getty ULAN/TGN

The WikidataMuseumFetcher.java class implements a sophisticated fallback strategy to link institutions to the Union List of Artist Names (ULAN) or the Thesaurus of Geographic Names (TGN).

  1. Primary Strategy: Query Wikidata for the museum. If a ULAN ID exists there, it is harvested.
  2. Fallback Strategy: If absent in Wikidata, the system queries the Getty SPARQL Endpoint (http://vocab.getty.edu/sparql) directly, searching for the museum's name using Lucene indexes.

This ensures that Romanian museums are linked to standard identifiers, facilitating global data interoperability. 


<http://arp.ro/resource/agent/Muzeul_Na_ional_Brukenthal___SIBIU>
    a                 arp:Museum;
    arp:gettyId       "500308991";
    arp:gettyUri      ulan:500308991;
    arp:imageLink     <http://commons.wikimedia.org/wiki/Special:FilePath/Palatul%20Brukenthal%20din%20Piata%20Mare.jpg>;
    arp:name          "Muzeul Național Brukenthal - SIBIU";
    arp:wikidataName  "Muzeul Brukenthal"@ro;
    arp:wikidataUri   <http://www.wikidata.org/entity/Q850207> .

8.3 Non-trivial SPARQL Queries and the ArP Query Service

To democratize access to the semantic data, the system exposes a dedicated web interface, the ArP Query Service. This Single Page Application (SPA) allows users to execute SPARQL queries against the integrated dataset without needing deep technical knowledge, thanks to a library of Predefined Queries injected dynamically from the backend via PredefinedQueryLoader.java.

The implemented queries go beyond simple data retrieval. They perform semantic aggregations and concept-based filtering by leveraging the SKOS alignment with Getty vocabularies established in the previous stages.

Query 1: Statistical Aggregation by Material (Getty AAT Alignment)

This query demonstrates the power of Linked Data. Instead of counting raw strings (where "ulei" and "oil" might be counted separately), it groups artworks by their resolved Getty AAT Code. It performs a join between the Artwork graph and the Vocabulary graph (skos:exactMatch). 


PREFIX arp:  <http://arp.ro/schema#>
PREFIX aat:  <http://vocab.getty.edu/aat/>
PREFIX skos: <http://www.w3.org/2004/02/skos/core#>
PREFIX prov: <http://www.w3.org/ns/prov#>

SELECT ?material_ro ?material_en (COUNT(DISTINCT ?artwork) AS ?numArtworks) ?aatCode
WHERE {
  # 1. Connect Creation Event to Artwork
  ?creation arp:materialsUsed ?material ;
            prov:generated ?artwork .

  # 2. Semantic Join: Map local string to Getty AAT Concept
  ?mapping skos:exactMatch ?aatCode ;
           rdfs:label ?labelRo .

  FILTER(LANG(?labelRo) = "ro")

  # 3. Fuzzy Matching: Ensure the raw string contains the vocabulary label
  FILTER(
    CONTAINS(LCASE(STR(?material)), LCASE(STR(?labelRo)))
  )

  # 4. Bilingual Label Handling
  BIND(STR(?labelRo) AS ?material_ro)
  OPTIONAL {
    ?mapping rdfs:label ?labelEn .
    FILTER(LANG(?labelEn) = "en")
  }
  BIND(IF(BOUND(?labelEn), STR(?labelEn), STR(?labelRo)) AS ?material_en)
}
GROUP BY ?material_ro ?material_en ?aatCode
ORDER BY DESC(?numArtworks)

Query 2: Institutional Analysis with External URIs

This query aggregates artwork holdings by Museum, but explicitly retrieves the external identifiers (Wikidata URI and Getty URI). This allows a researcher to immediately jump from the local dataset to the global knowledge base for any given museum. 


PREFIX arp:  <http://arp.ro/schema#>

SELECT ?museum ?museumName ?wikidataUri (COUNT(DISTINCT ?artwork) AS ?numArtworks) ?gettyUri
WHERE {
  # 1. Link Artwork to Location
  ?artwork a arp:Artwork ;
           arp:currentLocation ?museum .

  # 2. Retrieve Museum Metadata (including External Links)
  ?museum a arp:Museum .

  OPTIONAL { ?museum arp:name ?museumName }
  OPTIONAL { ?museum arp:gettyUri ?gettyUri }
  OPTIONAL { ?museum arp:wikidataUri ?wikidataUri }
}
GROUP BY ?museum ?museumName ?gettyUri ?wikidataUri
ORDER BY DESC(?numArtworks)

Query 3: Concept-Based Filtering (e.g., "Marble")

Unlike a keyword search, this query filters artworks based on a specific Getty Concept ID (e.g., aat:300011443 for "marble"). This ensures high recall, finding relevant artworks regardless of whether the local record uses the Romanian term "marmură" or the English "marble". 


SELECT DISTINCT ?artworkUri ?title ?material_ro ?material_en ?aatCode
WHERE {
  ?artwork a arp:Artwork ;
           arp:title ?title .

  ?creation arp:materialsUsed ?material ;
            prov:generated ?artwork .

  # Filter specifically by the Getty Concept ID for Marble
  ?mapping skos:exactMatch ?aatCode .
  FILTER(?aatCode = aat:300011443)

  # Retrieve labels for display
  ?mapping rdfs:label ?labelRo .
  FILTER(LANG(?labelRo) = "ro")

  BIND(?artwork AS ?artworkUri)
}
ORDER BY ?artwork

These queries are executed via the SparqlController, which ensures transactional read-access (Txn.executeRead) to the Apache Jena dataset, returning results in the standard JSON format. On the client side, the arp-query.js script (specifically the renderTable function) consumes this JSON to dynamically generate an HTML visualization. Crucially, the script inspects the metadata of each result binding; if a value is flagged as a uri, it is automatically rendered as a clickable hyperlink, while literals are displayed as plain text, ensuring a navigable experience consistent with Linked Data principles.

8.4 Conformance to Linked Data Principles

This project strictly adheres to the core principles of Linked Data as proposed by Tim Berners-Lee, ensuring interoperability, reusability, and semantic integration of cultural heritage data. The solution conforms to these principles through the following architectural decisions:

8.4.1 Use of HTTP URIs as Global Identifiers

All domain entities managed by the system—artworks, artists, cultural institutions, and creative activities—are identified using stable, HTTP-based URIs. These are consistent across the RDF datasets and serve as global identifiers.

For the national heritage dataset, the system explicitly reuses the authoritative identifiers provided by the Institute for Cultural Memory (CIMEC). These are extracted directly from the lido:lidoRecID XML tag (e.g., 0019570B3EDB...) to guarantee persistent identification.

Examples of identifiers generated by the system include:

  • Artworks: http://arp.ro/resource/artwork/{CIMEC_ID} (or {Wikidata_ID} for international works).
  • Agents: http://arp.ro/resource/agent/{Sanitized_Name}
  • Activities: http://arp.ro/resource/activity/creation_{CIMEC_ID} (linking the activity inextricably to the object).

Technical Note on URI Normalization: The placeholder {Sanitized_Name} refers to the output of a deterministic normalization algorithm implemented in the ETL pipeline. Since standard URIs cannot contain whitespace or specific reserved characters (as per RFC 3986), the system creates "slugified" identifiers by:

  1. Trimming leading/trailing whitespace;
  2. Replacing all non-alphanumeric characters (including diacritics and punctuation) with underscores (_);
  3. Collapsing consecutive underscores into a single separator.

For example, the raw input string "Grigorescu, Nicolae" is transformed into the valid URI segment Grigorescu_Nicolae. This ensures that the generated IRIs are syntactically valid and safe for transmission over HTTP without requiring aggressive percent-encoding (e.g., avoiding %20).

By adopting this URI scheme, the system ensures that each entity is uniquely identifiable and referenceable, a prerequisite for semantic linking.

8.4.2 Provision of Structured RDF Descriptions

Information is represented using the Resource Description Framework (RDF), encoded in Turtle (.ttl) format. The RDF descriptions capture descriptive metadata (titles, dimensions) and provenance (authorship, creation activities). The system leverages established vocabularies to ensure machine readability:

  • PROV-O: For provenance relationships (prov:wasAttributedTo, prov:wasGeneratedBy).
  • Dublin Core (DCT): For general descriptive metadata.
  • SKOS: For semantic mappings to Getty AAT.
  • ARP Namespace: A custom namespace to represent domain-specific concepts (e.g., arp:Registrar).

8.4.3 Access to Data via SPARQL Queries

While the HTTP URIs are primarily used as identifiers within the prototype's internal graph, the associated data is fully accessible via the SPARQL Endpoint exposed by the application.

The application acts as a semantic data layer, executing SPARQL queries over the Apache Jena models. This mechanism allows software agents (and the frontend SPA) to retrieve structured information by querying these URIs, fulfilling the intent of Linked Data access in a controlled environment.

8.4.4 Linking to External Knowledge Bases

The system establishes explicit semantic links between internal resources and the Linked Open Data cloud, specifically Wikidata and Getty Vocabularies. Entities are enriched using properties such as:

  • arp:wikidataUri – linking to Wikidata Entities.
  • skos:exactMatch – linking materials/categories to Getty AAT.
  • arp:gettyUri – linking museums/artists to ULAN/TGN.

By reusing external identifiers rather than duplicating information, the project promotes data reuse and context enrichment.

8.4.5 Summary of Alignment

Although public dereferenceability (content negotiation on the open web) is not implemented in the current academic prototype, the underlying data model and URI design fully support such an extension. Therefore, the solution conforms to Linked Data principles at the semantic and architectural level, meeting the requirements for a 4-Star Linked Data system (URIs, RDF, SPARQL) with strong elements of the 5th Star (External Linking).

9. Conclusion and Future Directions

The development of the ArP (Artwork Provenance) system demonstrates the practical viability of transitioning from siloed, legacy metadata to a dynamic, interconnected Knowledge Graph. By integrating robust Java technologies (Spring Boot, Apache Jena) with Semantic Web standards (RDF, SPARQL, PROV-O), the solution successfully bridges the gap between the national Romanian heritage and the global Linked Open Data cloud.

9.1 Overall Impression and Technical Impact

The project validates the "Hybrid Architecture" approach. The decision to decouple the semantic persistence layer (Jena TDB) from the presentation logic (Spring MVC) proved crucial for maintainability. Technically, the most significant achievement is the automated reconciliation pipeline (ETL) which successfully maps heterogeneous XML (LIDO) and JSON (Wikidata) sources into a unified ontology, enriched with Getty vocabulary links.

9.2 Impact on End Users and Usability

For the end-user (researchers, curators, and art enthusiasts), the system transforms the search experience. Instead of navigating disjointed databases, users can now perform semantic discovery navigating seamlessly from a Romanian painting to the international movement it belongs to, or tracing the provenance chain of an artwork through time. The ArP Query Service democratizes access to this complex data, allowing non-technical users to execute "non-trivial" queries (e.g., statistical aggregations by material or museum) via a user-friendly web interface.

9.3 Business Aspects and Sustainability

From a business perspective, the solution offers high value with low recurring costs:

  • Interoperability: By adhering to international standards (CIDOC-CRM alignment via PROV-O, Getty integration), the data becomes compatible with Europeana and other aggregators, increasing the visibility of Romanian heritage.
  • Cost-Efficiency: The use of open-source technologies (Java, RDF, SPARQL) eliminates licensing costs associated with proprietary museum management software.
  • Data Quality: The implementation of SHACL validation ensures that the repository remains a "single source of truth" with guaranteed structural integrity.

9.4 Further Directions of Development

While the current prototype meets all core requirements of a 4-Star Linked Data system, several avenues for future development are identified:

  • Full 5-Star Deployment: Implementing content negotiation to make URIs fully dereferenceable on the public web (serving HTML/RDF based on the `Accept` header).
  • Visual Analytics: Integrating a graph visualization library (e.g., Cytoscape.js or D3.js) to visually render the provenance node networks directly in the browser.
  • Crowdsourcing Module: Adding an authenticated "Curator Mode" where experts can manually correct mappings or add new relationships directly through the interface, turning the platform into a collaborative tool.
  • Federated Queries: Expanding the SPARQL endpoint to execute live federated queries against DBpedia or Europeana SPARQL endpoints, ensuring real-time data freshness.
